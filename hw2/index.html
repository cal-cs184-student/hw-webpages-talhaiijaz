Overview

In this homework I built a mesh editor from the ground up, starting with Bezier curves and surfaces and working up to full mesh manipulation and subdivision. Parts 1 and 2 implemented de Casteljau's algorithm, first for 2D curves, then extended to 3D tensor-product surfaces, which gave me a solid feel for how recursive linear interpolation can define arbitrarily smooth geometry from just a handful of control points.

Parts 3 through 5 shifted to triangle meshes and the half-edge data structure. Implementing area-weighted vertex normals (Part 3) was a good warm-up for traversing the half-edge structure, and it made the difference between faceted and smooth Phong shading immediately visible. Edge flips (Part 4) and edge splits (Part 5) were the real core of the assignment and they forced me to get comfortable with the full pointer-surgery involved in half-edge manipulation. The key insight was that these operations don't create or destroy geometric information; they just rearrange connectivity, and every single pointer (next, twin, vertex, edge, face) has to be accounted for or things silently break.

Part 6 brought everything together with Loop subdivision, which uses the split and flip primitives to repeatedly refine a coarse mesh into a smooth limit surface. The most interesting lesson here was how sensitive subdivision is to the initial mesh topology; the cube's asymmetric diagonal triangulation produced a lopsided blob until I pre-split each face into a symmetric four-triangle pattern. I also extended the subdivision to handle boundary meshes (EC1) and implemented an entirely different scheme, √3-subdivision (EC2), which inserts face centroids instead of edge midpoints and flips original edges to produce a 30°-rotated refinement.

Section I: Bezier Curves and Surfaces

Part 1: Bezier curves with 1D de Casteljau subdivision

De Casteljau's algorithm is an elegant recursive method for evaluating Bezier curves. Given \( n+1 \) control points defining a degree-\( n \) Bezier curve, the algorithm works by repeatedly performing linear interpolation between adjacent control points. At each step, given \( n \) points, we compute \( n-1 \) intermediate points using the formula:

\[ p'_i = (1 - t) \cdot p_i + t \cdot p_{i+1} \]

where \( t \in [0, 1] \) is the parameter at which we want to evaluate the curve. We apply this recursively, each step reduces the number of points by one, until we are left with a single point, which lies exactly on the Bezier curve at parameter \( t \).

I implemented this by writing evaluateStep, which performs exactly one level of this recursion. It takes in a vector of 2D points, loops through each adjacent pair, computes the lerp using the class member t, and pushes the result into a new vector. The framework then calls this function repeatedly to walk through all the levels.

Part 2: Bezier surfaces with separable 1D de Casteljau

To extend de Casteljau's algorithm from curves to surfaces, we take advantage of the fact that a Bezier surface is separable and we can evaluate it one dimension at a time. A Bezier patch is defined by an \( n \times n \) grid of control points, and we want to evaluate the surface at parameters \( (u, v) \).

The idea is simple: first, treat each row of control points as a Bezier curve and fully evaluate it at parameter \( u \) using de Casteljau's algorithm. This gives us \( n \) intermediate points: one per row. These \( n \) points then define a new Bezier curve in the other direction, which we evaluate at parameter \( v \) to get the final point on the surface.

I implemented this across three functions. evaluateStep is essentially the same as in Part 1, but operates on 3D points and takes t as a function parameter rather than a class member. evaluate1D calls evaluateStep in a loop until only one point remains, fully evaluating de Casteljau's algorithm for a single curve. Finally, evaluate ties it all together. It loops over each row of controlPoints, calls evaluate1D with parameter u on each row, collects the results, and then calls evaluate1D one more time with parameter v on the collected points.

Section II: Triangle Meshes and Half-Edge Data Structure

Part 3: Area-weighted vertex normals

To achieve smooth Phong shading instead of the default flat shading, we need to compute an approximate normal at each vertex. The idea is to take a weighted average of the normals of all the faces incident to that vertex, where each face's contribution is weighted by its area. The larger triangles should have more influence on the final normal direction.

I implemented this in Vertex::normal() by iterating around the vertex using the half-edge traversal pattern (h = h->twin()->next()). For each non-boundary face, I grab the three vertex positions of the triangle and compute the cross product of two of its edge vectors. The key insight is that the cross product's magnitude is proportional to twice the triangle's area, so it naturally gives us the area weighting for free and there's no need to compute the area separately. After summing up all the cross products, I normalize the result to get a unit normal.

Since this is a const member function, I used HalfedgeCIter throughout to avoid modifying the mesh.

Part 4: Edge flip

An edge flip is a local remeshing operation that takes two triangles sharing an edge and "rotates" that shared edge to connect the other pair of vertices instead. For triangles (a, b, c) and (c, b, d) sharing edge (b, c), the flip produces triangles (a, d, c) and (a, b, d) sharing edge (a, d).

I implemented this by first collecting all the mesh elements involved, the 6 halfedges, 4 vertices, 2 faces, and the shared edge. Then I used setNeighbors to systematically reassign every halfedge's pointers to reflect the new connectivity. The key to getting this right was drawing detailed before-and-after diagrams on paper with every element labeled, and then writing out each pointer assignment one by one. I also updated the vertex and face halfedge pointers to make sure none of them referenced a halfedge that had moved to a different face. Boundary edges are ignored since flipping them doesn't make geometric sense.

The main debugging strategy was simply being meticulous with the diagrams; I set every pointer explicitly, even the ones that didn't change, to avoid any missed updates. Flipping the same edge twice returns the mesh to its original state, which served as a good sanity check.

Part 5: Edge split

An edge split takes an edge shared by two triangles and inserts a new vertex at its midpoint, splitting the two original triangles into four. For triangles (a, b, c) and (c, b, d) sharing edge (b, c), the split places a new vertex \( m \) at the midpoint of (b, c) and creates four new triangles: (b, m, a), (m, c, a), (m, b, d), and (c, m, d).

I implemented this by first collecting all existing mesh elements, the 6 halfedges, 4 vertices, 2 faces, and the shared edge, just as in the flip operation. Then I allocated the new elements: 1 vertex (\( m \)), 3 new edges (\( m\text{-}a \), \( m\text{-}c \), \( m\text{-}d \)), 6 new halfedges, and 2 new faces (the original 2 faces are reused for 2 of the 4 resulting triangles). The most critical step was systematically rewiring all halfedge pointers using setNeighbors. I drew a detailed diagram labeling every element before and after the split, and then translated each pointer assignment directly from the diagram. I also set the isNew flags on edges: the two halves of the original split edge (\( b\text{-}m \) and \( m\text{-}c \)) are marked as old, while the two new edges crossing the original triangles (\( m\text{-}a \) and \( m\text{-}d \)) are marked as new, which is important for the subdivision routine in Part 6.

The debugging strategy was the same as for edge flip; drawing meticulous before-and-after diagrams with every halfedge, vertex, edge, and face labeled. I set every pointer explicitly, even the unchanged ones, to eliminate any possibility of a stale reference. Splitting and then flipping the same region served as a good sanity check: the mesh remained manifold with no holes or missing triangles.

Extra Credit: Boundary Edge Splits

For the extra credit, I extended splitEdge to support boundary edges. A boundary edge has one real face and one virtual boundary face (as described in the half-edge primer). Splitting it only makes geometric sense for the real-face side. We split that one triangle into two, and insert the new midpoint vertex into the boundary loop.

The main challenge was correctly updating the boundary face's halfedge loop. My approach avoids searching for the predecessor halfedge entirely: I reuse the original boundary halfedge \( h_3 \) (which previously went \( c \to b \)) as \( c \to m \), and insert a new halfedge \( m \to b \) right after it. Since \( h_3 \) is the same iterator, its predecessor's next pointer stays valid automatically. The result is 1 new vertex, 2 new edges, 1 new face, and 4 new halfedges, exactly half of the interior case, since there's only one real triangle to split.

Part 6: Loop subdivision for mesh upsampling

Loop subdivision upsamples a mesh by splitting every triangle into four and repositioning vertices using weighted averages. I implemented this in five steps: (1) compute updated positions for all existing vertices using the Loop rule, \( (1 - n \cdot u) \cdot \text{position} + u \cdot \text{neighborSum} \), where \( u = 3/16 \) for degree-3 vertices and \( u = 3/(8n) \) otherwise, storing them in Vertex::newPosition; (2) compute positions for new edge midpoint vertices using \( 3/8 \cdot (A + B) + 1/8 \cdot (C + D) \), storing in Edge::newPosition; (3) split every original edge (collected into a vector first to avoid an infinite loop); (4) flip any new edge that connects an old vertex to a new vertex, which corrects the connectivity into the proper 4-1 subdivision pattern; (5) copy all newPosition values into position. The key implementation detail was computing all positions before modifying the mesh, since the weighted averages depend on the original geometry.

After subdivision, sharp corners and edges get smoothed out and the mesh converges toward a smooth limit surface. This is visible in the icosahedron below, which goes from a faceted polyhedron to a nearly perfect sphere after a few rounds of subdivision.

When subdividing dae/cube.dae, the result becomes noticeably asymmetric. This happens because each face of the cube starts with only two triangles split along a single diagonal, and that diagonal introduces a preferred direction and the subdivision weights pull vertices differently along versus across the diagonal. Since all six faces have their diagonals oriented the same way, the asymmetry compounds with each subdivision level.

To fix this, I pre-processed the cube by splitting the diagonal edge on each face, giving every face a symmetric X-pattern of four triangles. This ensures that every vertex has the same degree and every face has a rotationally symmetric triangulation, so the subdivision weights pull equally in all directions. The result after subdivision is a much more symmetric, nearly spherical shape.

Extra Credit 1: Boundary Loop Subdivision

The standard Loop subdivision implementation only handles closed meshes so basically every edge has two real faces on either side. But meshes like the beetle have boundary edges (the wheel wells, hood opening, etc.) where an edge borders only one real face and one virtual boundary face. Without special treatment, applying the interior subdivision rules to boundary vertices produces catastrophic results: the weighting formula assumes a full ring of neighbors, so boundary vertices, which have fewer neighbors, get pulled wildly out of position, creating dramatic spikes.

To fix this, I added boundary-aware rules from the modified butterfly/cubic B-spline scheme described in the literature. For old boundary vertices, instead of the interior Loop rule, I use the cubic B-spline boundary rule: 1/8 * left + 3/4 * self + 1/8 * right, where left and right are the two boundary neighbors. For new boundary edge midpoints, I use a simple midpoint average: 1/2 * (A + B). I also skip the edge flip step for boundary edges, since flipping only makes sense for interior edges with two real faces. This required changes in both Step 1 (old vertex position computation) and Step 2 (edge midpoint computation) of the subdivision loop.

The difference is striking. Without boundary support, the beetle completely falls apart and boundary vertices shoot outward into long spikes because the interior weighting formula divides by a neighbor count that doesn't match reality. With boundary support, the mesh subdivides cleanly: the body smooths out, the wheel wells and hood opening maintain their shape, and the boundary edges stay exactly where they should be.

Extra Credit 2: √3-Subdivision

For the second extra credit, I implemented √3-subdivision (Kobbelt 2000), a fundamentally different subdivision scheme from Loop. While Loop performs a 1-to-4 edge split (inserting midpoints on edges), √3-subdivision performs a 1-to-3 face split and it inserts a new vertex at the centroid of every triangle, then flips all original edges to reconnect adjacent centroids. This produces a mesh that is rotated 30° each step, realigning after two steps (hence the name √3, since two applications yield a tri-adic split).

The smoothing rule for old vertices is: S(p) = (1 - αn) · p + αn · avg(neighbors), where αn = (4 - 2cos(2π/n)) / 9. A key practical difference is that √3 grows the triangle count by only 3× per step (vs 4× for Loop), giving a finer gradation of hierarchy levels. The limit surface is C² everywhere except at extraordinary vertices, matching Loop's smoothness guarantees.

Below I compare the two schemes on the same cube mesh. The top row shows Loop subdivision (L key), and the bottom row shows √3-subdivision (K key), at the same levels:

Both schemes converge to a rounded shape, but there are notable differences. The √3 result develops an egg-like asymmetry, a slight point or bulge on one side, more pronounced than Loop's asymmetry at the same level. This happens for the same root cause as Loop's cube asymmetry (the diagonal triangulation biases the smoothing), but the effect is amplified in √3 because the edge-flip step propagates the diagonal bias differently. The 30° rotation per step means the asymmetry doesn't simply align with the original diagonals and it compounds in a slightly different direction each step, producing the egg shape. Pre-splitting the cube faces symmetrically (as we did for Loop) would fix this here too.

Art Competition

For the art competition, I set out to model "CAL" block letters using cube primitives in Blender 5.0, with the goal of loading the model into the mesh editor and applying subdivision to smooth it into a stylized 3D logo.

Unfortunately, I ran into a compatibility issue: Blender 5.0 has completely removed the COLLADA (.dae) export option. COLLADA export was deprecated starting in Blender 4.0 and fully removed in later versions, so there is no built-in way to export a .dae file from the current version of Blender. I attempted to use a third-party converter to generate the .dae file, but the resulting file had several format incompatibilities with the mesh editor's COLLADA parser including material references to a missing material library and interleaved vertex/normal/texcoord inputs that caused the parser to misinterpret vertex indices. After multiple rounds of debugging and fixing the .dae file by hand, the mesh still failed to render correctly due to per-face vertex duplication in the exported geometry.

While the model itself never made it into the mesh editor for subdivision, the exercise was a useful lesson in file format compatibility and the challenges of working with export pipelines across different software versions.
